# All As Planned

All As Planned — Web-SaaS для маркетинговых команд: рабочие пространства и
проекты, контент-план, медиатека, согласования, подготовка видео и AI-инструменты
в одном кабинете. Публичная презентация доступна на `/`, защищённое приложение —
на `/app`, а системная панель появляется только у пользователя с `is_admin=true`.

Основные возможности Web:

- роли владельца, администратора, редактора, наблюдателя и клиента;
- контент-план, карточки материалов, версии, файлы и настраиваемые согласования;
- проектное файловое пространство: drag-and-drop, вложенные папки, поиск,
  переименование, перемещение и сохранение результатов AI;
- общий проектный чат, групповые и личные диалоги, обсуждения материалов,
  ответы, непрочитанные сообщения и вложения из файлов проекта;
- Chat Anywhere поверх любой страницы, защищённый мультимодальный просмотр,
  версии файлов и contextual review;
- живая карта проекта, редактор блок-схем и центр внимания с решениями,
  действиями, рисками и управленческими сводками;
- импорт YouTube, VK и Rutube, пакетная последовательная обработка роликов;
- несколько PNG/GIF/MOV/MP4-оверлеев с визуальным конструктором;
- AI-тексты, изображения, транскрибация и нарезка длинных видео на клипы;
- 14-дневный trial, тарифные лимиты, кредиты и платежи через ЮKassa;
- PostgreSQL-очередь, автоматическое удаление временных файлов, health/metrics и резервные копии.

## Файлы проектов и обсуждения

В разделе «Файлы» можно перетащить сразу несколько файлов, создать вложенные
папки, искать, переименовывать, перемещать и просматривать материалы. Сервер
проверяет не только расширение и MIME-тип, но и сигнатуру или структуру файла;
исполняемые файлы и замаскированный неподдерживаемый контент отклоняются. Лимит
одного файла задаётся `YT_LOADER_MAX_CONTENT_FILE_MB` (по умолчанию 250 МБ).

Результат AI-задания можно сохранить в выбранный проект и папку под понятным
именем: текст как Markdown/TXT, изображение как PNG, набор клипов как ZIP.
Пользовательские файлы не коммитятся в Git: метаданные хранятся в PostgreSQL,
а содержимое — в постоянном `server_data/`, поэтому резервировать нужно оба.

Раздел «Сообщения» объединяет автоматический общий чат проекта, пользовательские
группы, личные диалоги и контекстные обсуждения карточек контента. Доступ к чату
и вложениям ограничен участниками соответствующего workspace и проекта.

## Глубокое рабочее пространство

Workspace Depth V2 добавляет глобальное окно Chat Anywhere, мультимодальный
Asset Viewer, версии и contextual review, живую карту проекта, блок-схемы и
Decision Intelligence. Центр внимания извлекает сигналы только из общего чата
проекта и связанных сущностей; личные и закрытые групповые диалоги не становятся
источником автоматического анализа. AI-обогащение запускается явно, а без ключа

Модули выкатываются отдельными feature flags. Архитектура, API, модель угроз,
миграции и рекомендуемый порядок rollout описаны в `WORKSPACE_DEPTH_V2.md`.

## AI-студия Web

В разделе «AI-помощник» доступны генерация и редактура текстов, изображения и
нарезка загруженного в медиатеку видео на вертикальные клипы. Запросы выполняются
через общую очередь заданий и списывают кредиты только после успешного результата.
Ключ OpenAI хранится на сервере и никогда не передаётся браузеру.

Добавьте в `.env` минимум:

```env
OPENAI_API_KEY=sk-...
OPENAI_TEXT_MODEL=gpt-5.4-mini
OPENAI_IMAGE_MODEL=gpt-image-1.5
OPENAI_TRANSCRIPTION_MODEL=whisper-1
```

Новые аккаунты получают 14 дней пробного периода и 5 стартовых кредитов.
Коммерческие тарифы задаются миграцией: Creator — 1 490 ₽/месяц и 200 кредитов,
Studio — 3 990 ₽/месяц и 1 000 кредитов. Лимиты проектов, участников, хранилища
и очереди возвращаются `/api/billing/summary` и показываются на странице тарифа.

Для AI-нарезки сначала создайте материал в контент-плане и прикрепите к нему
видеофайл. FFmpeg извлечёт компактную аудиодорожку, OpenAI вернёт расшифровку с
таймкодами, а сервер выберет фрагменты и соберёт ZIP с MP4-клипами. Длинные записи
автоматически делятся на части меньше API-лимита 25 МБ, а их таймкоды склеиваются.

В репозитории также сохранено локальное desktop-приложение для загрузки видео по
списку ссылок. Его инструменты подготовки видео совместимы с Web-версией.

Что умеет:

- получает все ссылки из раздела Shorts по ссылке на YouTube-канал;
- создаёт несколько вариантов видео с выбранными изображениями или анимированными
  оверлеями и раскладывает их по отдельным папкам;
- вставка ссылок через `Ctrl+V` или кнопку "Вставить из буфера";
- загрузка списка ссылок из `.txt`;
- выбор папки сохранения;
- выбор разрешения скачивания: 1920x1080 по умолчанию, 2K или 4K при необходимости;
- автоподбор cookies для YouTube и VK;
- последовательная загрузка видео в `mp4`;
- три режима метаданных: сохранение, стандартная очистка и экспериментальная подмена.

## Формат ссылок

Поддерживаются обычные ссылки, по одной в строке.

## Импорт Shorts с канала

1. Вставь ссылку на канал в поле «Импорт Shorts с YouTube-канала» — например,
   `https://www.youtube.com/@имя_канала`.
2. Нажми «Получить ссылки Shorts» или клавишу `Enter`.
3. Приложение добавит найденные ролики в общий список ссылок. Уже добавленные
   ссылки будут пропущены.
4. Проверь список и нажми «Скачать». Десктопная версия временно не получает
   описания и теги и не создаёт CSV — импортируется только полный список ссылок.

## Логотип поверх видео

1. В блоке «Изображение/анимация поверх видео» выбери один или несколько файлов.
   Поддерживаются PNG, JPG, GIF, APNG, WebP, MOV, MP4, M4V, WebM, MKV, AVI,
   MPEG и другие форматы, которые установленный FFmpeg распознаёт как видео.
2. Укажи непрозрачность: например, `35%` означает заметный, но полупрозрачный
   логотип.
3. Укажи ширину относительно кадра. Значение `22%` подходит для большинства
   вертикальных Shorts.
4. В конструкторе 9:16 перетащи оверлей в нужное место. Угловой маркер меняет размер;
   стрелки клавиатуры двигают его на 1%, а `Shift+стрелка` — на 5%. При нескольких оверлеях щёлкни по нужному
   файлу в списке, чтобы посмотреть его; общие X/Y, ширина и прозрачность применяются ко всем вариантам.
5. Нажми «Скачать». Сначала приложение скачает все доступные исходные видео.
   После завершения всей загрузочной очереди начнётся отдельный этап создания
   вариантов с выбранными оверлеями.

В выбранной папке автоматически создаётся по одной подпапке на каждый файл.
Например, для `cat.gif` и `promo.gif` результат будет сохранён в `cat\` и
`promo\`. Если имена совпадают, используются безопасные суффиксы: `promo_2`,
`promo_3` и так далее. Исходники хранятся во временной служебной папке до начала
второго этапа. Каждый исходник удаляется только после успешного создания всех
его вариантов; при ошибке он остаётся для повторной обработки.

Для максимальной скорости приложение использует NVIDIA NVENC, если он доступен.
Если аппаратное кодирование не запускается, автоматически применяется быстрый
режим `libx264`. Исходный файл заменяется только после успешной обработки.

## Режимы метаданных

Одинаковые режимы доступны в desktop и web:

- **Без очистки** — FFmpeg сохраняет доступные исходные поля контейнера;
- **Стандартная очистка** — режим по умолчанию: исходные глобальные и потоковые
  метаданные и главы не копируются в итоговый MP4;
- **Экспериментальная подмена** — сначала выполняется стандартная очистка, затем
  записывается стабильный синтетический профиль Apple/iPhone, версия iOS и дата.
  Координаты GPS не добавляются, пиксели и звук ради обхода распознавания не меняются.

Если оверлей не выбран, очистка выполняется быстрым пересбором контейнера с
`-c copy`, без повторного кодирования и потери качества. При наложении оверлея
режим применяется в том же FFmpeg-проходе, поэтому отдельного тяжёлого этапа нет.
Подмена метаданных не делает ролик анонимным и не гарантирует, что площадки не
сопоставят его с оригиналом по изображению или звуку.

### Быстрый запуск с NVIDIA CUDA/NVENC

Само приложение запускается обычной командой, а доступный кодировщик NVIDIA
`h264_nvenc` выбирается автоматически. CUDA Toolkit для этого не требуется:
достаточно актуального драйвера NVIDIA и сборки FFmpeg с NVENC.

Проверка видеокарты и кодировщика в PowerShell:

```powershell
nvidia-smi
.\ffmpeg.exe -hide_banner -encoders | Select-String h264_nvenc
```

Если `ffmpeg.exe` установлен через `PATH`, используй вместо второй команды:

```powershell
ffmpeg -hide_banner -encoders | Select-String h264_nvenc
```

Запуск десктопного UI:

```powershell
C:\ProgramData\anaconda3\python.exe app.py
```

Во время обработки в логе появится `(NVENC)`. Если NVENC недоступен или завершится
ошибкой, приложение автоматически повторит кодирование на CPU.

Поддерживаются адреса каналов вида `youtube.com/@handle`,
`youtube.com/channel/UC...`, `youtube.com/c/...` и `youtube.com/user/...`.

## Веб-сервис для собственного сервера

В проект добавлена отдельная серверная версия. Она не требует GPU и выполняет
тяжёлые операции по одной в фоновой очереди. Веб-интерфейс позволяет:

- импортировать YouTube Shorts, VK Video/Clips и каналы/плейлисты Rutube вместе с описаниями и тегами;
- выбрать, сколько видео получить: 10, 25, 50, 100, 200 или все;
- скачать общий CSV;
- отдельно выгрузить TXT с тегами и описанием каждого ролика;
- отдельно подготовить и скачать MP4 каждого ролика;
- выбрать галочками несколько Shorts и последовательно подготовить их одной очередью;
- вставить ссылку на конкретное видео YouTube, VK или Rutube и обработать его без импорта канала;
- сохранить найденный ролик в контент-план выбранного проекта без повторных дублей;
- перед обработкой загрузить до 10 изображений или анимаций/видео. Исходник скачивается один раз,
  для каждого оверлея создаётся отдельный вариант, а результат выдаётся ZIP-архивом с отдельной папкой на каждый файл.
  Веб-версия принимает до 256 МБ на оверлей и пишет загрузку на диск потоково,
  не удерживая файл целиком в RAM; лимит можно изменить переменной `YT_LOADER_MAX_OVERLAY_MB`;
- в реальном времени разместить оверлей в вертикальном кадре 9:16: перетащивать его
  мышью или пальцем, менять размер за угловой маркер и видеть прозрачность до запуска FFmpeg;
- выбрать один из трёх режимов обработки метаданных перед подготовкой ролика.

#### Первое обновление старой Basic Auth версии

Переход со старого контейнера без PostgreSQL выполняй отдельно от обычных обновлений.
Сначала сохрани текущий commit и проверь, что tracked-файлы на сервере не изменены:

```bash
cd /opt/yt-loader
git status --short
git rev-parse HEAD
```

Затем получи release, но пока не перезапускай контейнер:

```bash
git fetch origin
git pull --ff-only origin main
cp .env .env.before-saas
chmod 600 .env .env.before-saas
nano .env
```

Добавь в `.env` как минимум `POSTGRES_PASSWORD`, публичный HTTPS URL, допустимые host и
явный режим email-подтверждения. Для первого запуска без готового SMTP временно укажи
`YT_LOADER_REQUIRE_EMAIL_VERIFICATION=false`; перед публичной регистрацией SMTP и
подтверждение обязательно нужно включить.

```dotenv
POSTGRES_PASSWORD=replace-with-url-safe-openssl-value
YT_LOADER_PUBLIC_BASE_URL=https://shorts.example.com
YT_LOADER_SECURE_COOKIES=true
YT_LOADER_ALLOWED_HOSTS=shorts.example.com,localhost,127.0.0.1
YT_LOADER_REQUIRE_EMAIL_VERIFICATION=false
```

Сгенерировать URL-safe пароль PostgreSQL и выполнить контролируемое переключение:

```bash
openssl rand -base64 36 | tr -dc 'A-Za-z0-9_.~-' | head -c 40; echo
chmod +x deploy/first-saas-upgrade.sh
./deploy/first-saas-upgrade.sh
```

Скрипт не меняет Git и не выводит секреты. Он создаёт root-only копию `.env`, `cookies`
и `server_data` в `/var/backups/yt-loader-pre-saas/`, собирает образ при ещё работающей
старой версии, применяет миграции PostgreSQL и только затем заменяет контейнер приложения.

Перед первым запуском создай пароль PostgreSQL и запиши его в
`POSTGRES_PASSWORD` внутри `.env`:

```bash
openssl rand -hex 32
```

PostgreSQL не публикует порт наружу и доступен только контейнерам во внутренней сети.
Сервис `migrate` автоматически выполняет `alembic upgrade head` до запуска web-приложения.
Проверить готовность базы и фоновых worker можно через `/api/health`: исправная версия
возвращает `{"status":"ok","database":"ok","workers":"ok"}`. При неисправности
endpoint отвечает HTTP 503, поэтому Docker healthcheck перезапустит нездоровый сервис.

Очередь заданий также хранится в PostgreSQL. Worker атомарно арендует задание и
продлевает аренду во время FFmpeg/yt-dlp обработки. Если контейнер аварийно
перезапустился, после истечения `YT_LOADER_JOB_LEASE_SECONDS` задание автоматически
вернётся в очередь; после трёх сорванных попыток оно завершится ошибкой, а не будет
перезапускаться бесконечно. Для обычного CPU-сервера оставь значения `120` секунд и
один процесс Uvicorn.

При `YT_LOADER_ALLOW_REGISTRATION=true` пользователь создаёт аккаунт прямо на странице.
Для публичной регистрации оставь `YT_LOADER_REQUIRE_EMAIL_VERIFICATION=true` и настрой
SMTP. Без рабочей почтовой доставки сервис намеренно не создаёт аккаунты, которые затем
невозможно подтвердить или восстановить:

```dotenv
YT_LOADER_PUBLIC_BASE_URL=https://shorts.example.com
YT_LOADER_REQUIRE_EMAIL_VERIFICATION=true
SMTP_HOST=smtp.example.com
SMTP_PORT=587
SMTP_SECURITY=starttls
SMTP_USERNAME=mailer@example.com
SMTP_PASSWORD=replace-with-smtp-password
SMTP_FROM_EMAIL=All As Planned <no-reply@allasplanned.ru>
```

Ссылка подтверждения действует 24 часа, ссылка сброса пароля — 1 час; в базе хранится
только SHA-256 хеш одноразового токена. Смена или восстановление пароля отзывает старые
сессии. Для закрытой локальной установки без почты можно явно поставить
`YT_LOADER_REQUIRE_EMAIL_VERIFICATION=false`.

Для создания или повышения администратора выполни интерактивную команду — пароль
не попадёт в историю shell:

```bash
docker compose exec yt-loader python manage_users.py create-admin --email admin@example.com
```

Для закрытого сервиса после создания администратора установи в `.env`
`YT_LOADER_ALLOW_REGISTRATION=false` и выполни `docker compose up -d`.

Новый аккаунт получает 5 пробных кредитов; значение задаётся через
`YT_LOADER_SIGNUP_CREDITS`. Импорт канала резервирует 1 кредит за каждые начатые
100 запрошенных Shorts (вариант «все» временно резервирует 10), но после импорта
списывается только стоимость фактически найденного количества. Подготовка видео стоит
по 1 кредиту за каждый создаваемый вариант оверлея. При ошибке worker весь резерв
возвращается на баланс. Начислить кредиты вручную:

```bash
docker compose exec yt-loader python manage_users.py grant-credits \
  --email user@example.com --amount 100 --reason "Тестовое начисление"
```

Проверить, совпадают ли быстрые счётчики с неизменяемым журналом и резервами заданий:

```bash
docker compose exec yt-loader python manage_users.py audit-credits
```

В каталоге пока заданы предварительные тарифы: пробный, Creator (100 кредитов) и
Studio (500 кредитов). Цены являются конфигурацией продукта перед подключением
платёжного провайдера на следующем этапе и могут быть изменены отдельной миграцией.

Сервис будет доступен только локально на сервере по `127.0.0.1:8000`. Для
публичного доступа скопируй `deploy/nginx.conf` в конфигурацию Nginx, замени
`YOUR_DOMAIN`, включи сайт и выпусти HTTPS-сертификат.

```bash
sudo apt update
sudo apt install -y nginx certbot python3-certbot-nginx
sudo cp deploy/nginx.conf /etc/nginx/sites-available/yt-loader
sudo ln -s /etc/nginx/sites-available/yt-loader /etc/nginx/sites-enabled/yt-loader
sudo nginx -t
sudo systemctl reload nginx
sudo certbot --nginx -d YOUR_DOMAIN
```

Если домена пока нет, укажи в Nginx `server_name _;`, пропусти Certbot и
открывай панель по IP-адресу через HTTP. В таком временном варианте установи
`YT_LOADER_SECURE_COOKIES=false`; после включения HTTPS обязательно верни `true`.
Логин и пароль теперь принадлежат аккаунту в базе, а старый Basic Auth включается
только явно через `YT_LOADER_LEGACY_BASIC_AUTH=true`.

Данные очереди, метаданные и готовые файлы сохраняются в `server_data/`.
Исходный MP4 заменяется обработанной версией и отдельно не хранится. После первого нажатия
«Скачать MP4» таймер удаления запускается после завершения передачи файла, по умолчанию 15 минут. Срок задаётся
`YT_LOADER_AFTER_DOWNLOAD_MINUTES`. Если файл не скачивали, он удаляется через `YT_LOADER_READY_HOURS` часов
(по умолчанию 24). Кнопка «Удалить сейчас» удаляет готовый файл сразу. Общая очистка
остальных данных регулируется `YT_LOADER_RETENTION_HOURS`. Cookies YouTube, VK и
Rutube монтируются соответственно из `cookies/www.youtube.com_cookies.txt`,
`cookies/vk.com_cookies.txt` и `cookies/rutube.ru_cookies.txt` и не попадают в Git.
Отсутствующий cookie-файл просто не используется. Контейнеру нужен доступ на запись
к папке `cookies`, поскольку `yt-dlp` может обновлять cookies во время работы.

Для установки без Docker подготовлены `requirements-server.txt` и unit-файл
`deploy/yt-loader.service`. При ручной установке также нужны системные пакеты
`ffmpeg` и `nodejs`.

### Резервная копия PostgreSQL

```bash
docker compose exec -T postgres pg_dump -U yt_loader yt_loader > yt_loader_backup.sql
```

Для регулярной проверяемой копии в custom-формате используй готовый скрипт. Он сначала
проверяет архив через `pg_restore --list`, атомарно переименовывает готовый файл и удаляет
локальные копии старше заданного срока:

```bash
chmod +x deploy/backup-postgres.sh deploy/restore-postgres.sh
BACKUP_DIR=/var/backups/yt-loader BACKUP_RETENTION_DAYS=14 ./deploy/backup-postgres.sh
```

Добавь эту команду в cron и обязательно отправляй копии на другое физическое хранилище.
Пример ежедневного запуска в 03:15:

```cron
15 3 * * * cd /opt/yt-loader && BACKUP_DIR=/var/backups/yt-loader ./deploy/backup-postgres.sh >> /var/log/yt-loader-backup.log 2>&1
```

Восстановление останавливает приложение, заменяет текущую базу, применяет миграции и
запускает обе финансовые сверки:

```bash
RESTORE_CONFIRM=yt_loader ./deploy/restore-postgres.sh /var/backups/yt-loader/yt_loader_YYYYMMDDTHHMMSSZ.dump
```

Храни резервные копии вне сервера с приложением. PostgreSQL содержит аккаунты,
сессии, задания, параметры очереди и реестр файлов. Само видео остаётся в
`server_data/`, поэтому для полного восстановления сервиса резервируй и базу, и этот
каталог. Старые JSON-задания при первом запуске импортируются в PostgreSQL только как
история; незавершённое старое задание нельзя продолжить, потому что прежний формат не
сохранял его входные параметры.

Для одной ежедневной команды, сохраняющей и PostgreSQL, и `server_data`, используй:

```bash
chmod +x deploy/backup-data.sh
YT_LOADER_BACKUP_DIR=/var/backups/yt-loader ./deploy/backup-data.sh
```

Скрипт создаёт два файла с одинаковой UTC-меткой: `postgres-*.dump` и
`files-*.tar.gz`. Локальное хранение по умолчанию — 14 дней; внешний перенос копий
нужно настроить отдельно на уровне сервера или объектного хранилища.

### Подключение оплаты через ЮKassa

Оплата включается только при одновременном наличии публичного HTTPS-адреса, `shopId` и
секретного ключа. Сначала используй тестовый магазин ЮKassa и тестовые банковские карты.
В `.env` на сервере укажи:

```dotenv
YT_LOADER_PUBLIC_BASE_URL=https://shorts.example.com
YOOKASSA_SHOP_ID=your-shop-id
YOOKASSA_SECRET_KEY=your-secret-key
YOOKASSA_WEBHOOK_ENFORCE_IP=true
```

Файл с секретами не должен попадать в Git и должен быть доступен только владельцу:

```bash
chmod 600 .env
docker compose up -d --build
docker compose logs migrate
docker compose logs -f yt-loader
```

В личном кабинете ЮKassa создай webhook на адрес:

```text
https://shorts.example.com/api/payments/yookassa/webhook
```

Подпиши его на события `payment.succeeded` и `payment.canceled`. Nginx принимает этот
endpoint отдельно, только методом POST и с лимитом тела 128 КБ. Приложение дополнительно
проверяет исходный IP по списку сетей ЮKassa и всегда запрашивает текущее состояние
платежа через API провайдера: статус и сумма из входящего JSON сами по себе не считаются
доказательством оплаты.

По умолчанию отправка чека из API выключена. Требования 54-ФЗ, состав чека и ставку НДС
нужно согласовать с бухгалтером под конкретную организацию и настройки магазина. Если
чек должен формировать этот сервис, включи режим только после такой настройки:

```dotenv
YOOKASSA_RECEIPT_ENABLED=true
YOOKASSA_VAT_CODE=1
```

После тестовой оплаты проверь баланс в интерфейсе и выполни сверку:

```bash
docker compose exec yt-loader python manage_users.py audit-payments
docker compose exec yt-loader python manage_users.py audit-credits
```

Обе команды должны завершиться с кодом `0` и без строк `MISMATCH`. Для диагностики
смотри журнал контейнера и таблицы `payments`/`webhook_events`; секретный ключ и полное
тело ответа провайдера туда не сохраняются. При утечке ключа немедленно перевыпусти его
в кабинете ЮKassa и перезапусти контейнер.

Перед переходом на боевой магазин обязательно проверь: успешную оплату, отмену,
повторную доставку одного webhook, возврат пользователя на сайт раньше webhook,
автопродление с сохранённым способом оплаты и сценарий отказа банка. Реальный запуск
также требует публичной оферты, политики конфиденциальности, правил возврата и
контактных данных продавца; данный код не подменяет юридическую проверку.

Полный release gate перед открытием регистрации находится в
[`deploy/production-checklist.md`](deploy/production-checklist.md). Пункты, зависящие от
боевого домена, SMTP, ЮKassa, бухгалтерии и юридических документов, нельзя подтвердить
только unit-тестами проекта.
