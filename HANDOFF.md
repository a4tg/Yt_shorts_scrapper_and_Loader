## Как продолжить на другом компьютере

```bash
git clone https://github.com/a4tg/Yt_shorts_scrapper_and_Loader.git
cd Yt_shorts_scrapper_and_Loader
```

После открытия проекта передайте Codex такую задачу:

> Прочитай HANDOFF.md, README.md, deploy/production-checklist.md и последние
> коммиты Git. Продолжи с первого незавершённого этапа. Не изменяй и не
> коммить пользовательский каталог reports/. После каждого этапа дай короткий
> отчёт и объясняй команды по ходу работы.

Перед любыми изменениями проверить состояние:

```bash
git status --short
git log -10 --oneline
```

Каталог `reports/` локальный, принадлежит владельцу проекта и не должен
добавляться в коммиты без отдельного явного запроса.

## Продукт и направление

All As Planned — Web-SaaS для маркетологов, контент-команд и агентств. Цель —
объединить в одном рабочем пространстве контент-план, медиатеку, материалы,
согласования, подготовку видео и AI-инструменты. Идеология продукта: на сервис
можно положиться в вопросах сохранности и неприкосновенности данных, скорости
работы и предсказуемости результата.

Первый рынок — Россия, затем возможен worldwide-запуск. Стартовый рекламный
бюджет обсуждался в размере около 10 000 рублей. Желаемый доход владельца — не
менее 250 000 рублей в месяц, но экономика ещё должна быть подтверждена реальными
конверсиями и стоимостью обработки.

## Резервное копирование

Этап резервного копирования завершён и проверен восстановлением.

- локальные архивы: `/var/backups/yt-loader`;
- ежедневный cron: `/etc/cron.d/yt-loader-backup`;
- запуск: ежедневно в `03:15` по времени production-сервера;
- локальный скрипт проекта: `deploy/backup-data.sh`;
- внешняя зашифрованная копия отправляется Restic на отдельный зарубежный
  сервер через SSH alias `aap-backup-storage`;
- активный Restic-репозиторий имеет путь
  `sftp:aap-backup-storage:/srv/aap-backups/restic-v2`;
- пароль читается из `/root/.config/restic/aap-password`, права `600`;
- обёртка: `/usr/local/sbin/aap-restic-backup`;
- политика: 14 ежедневных снимков;
- журнал: `/var/log/yt-loader-backup.log`;
- logrotate: `/etc/logrotate.d/yt-loader-backup`, 14 сжатых ротаций.

Подтверждённые проверки:

- неинтерактивное открытие нового репозитория работает;
- полный снимок создан и `restic check` завершился без ошибок;
- тестовый `restic restore latest` восстановил файлы;
- файловый `tar.gz` успешно прочитан;
- дамп PostgreSQL проверен через `pg_restore` из `postgres:17-alpine`;
- тестовая директория после проверки удалена.

Системный `pg_restore` production-хоста старее формата дампа PostgreSQL 17.
Для проверки дампа использовать образ `postgres:17-alpine`, иначе возникает
ошибка `unsupported version (1.16) in file header`.

Никогда не публиковать Restic-пароль. Он должен дополнительно храниться вне обоих
серверов в защищённом хранилище. Старый тестовый Restic-репозиторий не является
активным; не удалять его без отдельной проверки и явного решения владельца.

## AI и видео: дальнейшее направление

- Нарезку длинных видео развивать с учётом репозитория
  `a4tg/AI_shorts_Cutter`, но не переносить код вслепую: сначала проверить
  лицензию, качество сегментации, зависимости и совместимость с очередью Web.
- Для первого запуска выгоднее использовать внешние API для текстов,
  изображений и транскрибации, а FFmpeg оставить на собственном CPU-сервере.
- Тяжёлую генерацию изображений на production без GPU не размещать.
- Стоимость AI должна списываться через кредиты только после успешного задания;
  реальные лимиты и тарифы уточнить после измерения себестоимости.
- Максимальное рабочее разрешение на первом этапе — 1080p, параллельных тяжёлых
  обработок на текущем сервере должно быть не более пяти и только после
  нагрузочной проверки.

## Правила безопасного продолжения

- Никогда не выводить и не коммитить `.env`, SMTP-пароль, PostgreSQL-пароль,
  API-ключи, cookie-файлы, SSH private keys и Restic-пароль.
- В переписке ранее могли появляться старые секреты. Не повторять их; при
  сомнении перевыпускать.
- Перед изменением Nginx выполнять `nginx -t`, перед Docker-обновлением —
  `docker compose config --quiet`.
- Перед рискованной миграцией создавать свежий локальный и внешний снимок.
- Не удалять пользовательские файлы и не выполнять destructive Git-команды.
- Коммиты должны содержать только относящиеся к этапу изменения.
