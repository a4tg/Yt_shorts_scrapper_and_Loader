# All As Planned: передача контекста

Актуально на 16 июля 2026 года. Этот файл нужен для продолжения разработки в
новой сессии Codex или на другом компьютере. Он намеренно не содержит паролей,
API-ключей, cookie-файлов и значений production `.env`.

## Как продолжить на другом компьютере

```bash
git clone https://github.com/a4tg/Yt_shorts_scrapper_and_Loader.git
cd Yt_shorts_scrapper_and_Loader
```

После открытия проекта передайте Codex такую задачу:

> Прочитай HANDOFF.md, README.md, deploy/production-checklist.md и последние
> коммиты Git. Продолжи с первого незавершённого этапа. Не изменяй и не
> коммить пользовательский каталог reports/. После каждого этапа дай короткий
> отчёт и объясняй команды по ходу работы.

Перед любыми изменениями проверить состояние:

```bash
git status --short
git log -10 --oneline
```

Каталог `reports/` локальный, принадлежит владельцу проекта и не должен
добавляться в коммиты без отдельного явного запроса.

## Продукт и направление

All As Planned — Web-SaaS для маркетологов, контент-команд и агентств. Цель —
объединить в одном рабочем пространстве контент-план, медиатеку, материалы,
согласования, подготовку видео и AI-инструменты. Идеология продукта: на сервис
можно положиться в вопросах сохранности и неприкосновенности данных, скорости
работы и предсказуемости результата.

Первый рынок — Россия, затем возможен worldwide-запуск. Стартовый рекламный
бюджет обсуждался в размере около 10 000 рублей. Желаемый доход владельца — не
менее 250 000 рублей в месяц, но экономика ещё должна быть подтверждена реальными
конверсиями и стоимостью обработки.

- публичный лендинг `/` и защищённое приложение `/app`;

- контент-план, материалы, версии, файлы и настраиваемое согласование;

  проектные папки, поиск, переименование и перемещение файлов;
- сохранение текстов, изображений и ZIP с клипами из AI-заданий в проект;
- общий проектный чат, группы, личные и контекстные диалоги, ответы,
  непрочитанные сообщения и вложения из файлов проекта;
- импорт YouTube Shorts, а также источники VK и Rutube;
- загрузка и последовательная пакетная обработка выбранных видео;
- получение ссылок, описаний, тегов и метаданных;
- несколько PNG/GIF/MOV/MP4-оверлеев на одно задание;
- конструктор оверлеев поверх кадра реального видео: позиция, размер до 100%,
  прозрачность и предпросмотр;
- три режима работы с метаданными;
- автоматическое удаление временных исходников и результатов по сроку хранения;
- AI-студия: тексты, изображения, транскрибация и нарезка длинного видео;
- тарифы, trial, кредиты, платёжная модель и интеграционный слой ЮKassa;
- PostgreSQL-очередь заданий, health/readiness/metrics и административные
  инструменты;
- адаптивный интерфейс, бренд All As Planned, анимации лендинга и рабочих
  процессов, фоновые частицы и поддержка `prefers-reduced-motion`.

Desktop-версия сохранена в репозитории. Её видеоинструменты должны оставаться
функционально совместимыми с Web, но текущий коммерческий приоритет — Web.

- основной адрес: `https://allasplanned.ru`;
- `www.allasplanned.ru` также включён в TLS-сертификат;

  IPv6;

- SMTP через почтовый ящик домена работает;
- реальная регистрация, письмо подтверждения и восстановление пароля проверены
  end-to-end;
- production `.env` имеет права `600` и не хранится в Git.

Почтовые адреса проекта: `no-reply@allasplanned.ru` и
`support@allasplanned.ru`. Значения SMTP-пароля не фиксировать в документации и
Git.

### Обновление production из Git

Сначала убедиться, что на сервере нет неожиданных локальных изменений:

```bash
cd /opt/yt-loader
git status --short
git pull --ff-only origin main
docker compose up -d --build
docker compose ps
curl -fsS https://allasplanned.ru/api/health
curl -fsS https://allasplanned.ru/api/auth/config
```

Нельзя перезаписывать production `.env`, cookie-файлы и пользовательские данные
при обычном обновлении.

### Полезная диагностика production

```bash
cd /opt/yt-loader
docker compose ps
docker compose logs --tail=200 yt-loader
docker compose logs --tail=100 postgres
curl -fsS https://allasplanned.ru/api/health
df -h /
```

## Резервное копирование

Этап резервного копирования завершён и проверен восстановлением.

- локальные архивы: `/var/backups/yt-loader`;
- ежедневный cron: `/etc/cron.d/yt-loader-backup`;
- запуск: ежедневно в `03:15` по времени production-сервера;
- локальный скрипт проекта: `deploy/backup-data.sh`;
- внешняя зашифрованная копия отправляется Restic на отдельный зарубежный
  сервер через SSH alias `aap-backup-storage`;
- активный Restic-репозиторий имеет путь
  `sftp:aap-backup-storage:/srv/aap-backups/restic-v2`;
- пароль читается из `/root/.config/restic/aap-password`, права `600`;
- обёртка: `/usr/local/sbin/aap-restic-backup`;
- политика: 14 ежедневных снимков;
- журнал: `/var/log/yt-loader-backup.log`;
- logrotate: `/etc/logrotate.d/yt-loader-backup`, 14 сжатых ротаций.

Подтверждённые проверки:

- неинтерактивное открытие нового репозитория работает;
- полный снимок создан и `restic check` завершился без ошибок;
- тестовый `restic restore latest` восстановил файлы;
- файловый `tar.gz` успешно прочитан;
- дамп PostgreSQL проверен через `pg_restore` из `postgres:17-alpine`;
- тестовая директория после проверки удалена.

Системный `pg_restore` production-хоста старее формата дампа PostgreSQL 17.
Для проверки дампа использовать образ `postgres:17-alpine`, иначе возникает
ошибка `unsupported version (1.16) in file header`.

Никогда не публиковать Restic-пароль. Он должен дополнительно храниться вне обоих
серверов в защищённом хранилище. Старый тестовый Restic-репозиторий не является
активным; не удалять его без отдельной проверки и явного решения владельца.

## Что блокирует нормальный коммерческий запуск

Следующий рабочий этап — мониторинг. Рекомендуемая последовательность:

1. На отдельном зарубежном сервере проверять
   `https://allasplanned.ru/api/health`, чтобы обнаруживать полное отключение
   production.
2. На production контролировать свободное место, Docker health, состояние
   PostgreSQL/worker и свежесть последнего Restic-снимка.
3. Настроить уведомления владельцу на email и в Telegram.
4. Провести нагрузочный тест на типичном ролике и максимальном разрешённом
   пакете; определить реальную очередь и себестоимость CPU/диска/API.
5. Подключить и end-to-end проверить тестовую, затем боевую ЮKassa: успешный и
   отменённый платёж, повтор webhook, возврат, аудит платежей и кредитов.
6. Согласовать чеки, НДС и 54-ФЗ с бухгалтером; опубликовать оферту, политику
   конфиденциальности, правила возврата и реквизиты продавца.
7. Проверить пользовательские лимиты, очистку данных, поддержку и сценарии
   закрытой беты на нескольких реальных командах.
8. Только после release gate открывать публичный коммерческий трафик.

Полный список проверок находится в `deploy/production-checklist.md`. Наличие
функции в коде не означает, что её production-интеграция уже проверена.

Файловое пространство и чаты завершены и проверены локально, но ещё не считаются
проверенными на production. Перед выкладкой нужен свежий снимок PostgreSQL и
`server_data`, миграция до `g2b3c4d5e6f7`, проверка свободного места и закрытая
end-to-end проверка изоляции проектов, загрузки, восстановления файлов и сообщений.

## AI и видео: дальнейшее направление

- Нарезку длинных видео развивать с учётом репозитория
  `a4tg/AI_shorts_Cutter`, но не переносить код вслепую: сначала проверить
  лицензию, качество сегментации, зависимости и совместимость с очередью Web.
- Для первого запуска выгоднее использовать внешние API для текстов,
  изображений и транскрибации, а FFmpeg оставить на собственном CPU-сервере.
- Тяжёлую генерацию изображений на production без GPU не размещать.
- Стоимость AI должна списываться через кредиты только после успешного задания;
  реальные лимиты и тарифы уточнить после измерения себестоимости.
- Максимальное рабочее разрешение на первом этапе — 1080p, параллельных тяжёлых
  обработок на текущем сервере должно быть не более пяти и только после
  нагрузочной проверки.

## Правила безопасного продолжения

- Никогда не выводить и не коммитить `.env`, SMTP-пароль, PostgreSQL-пароль,
  API-ключи, cookie-файлы, SSH private keys и Restic-пароль.
- В переписке ранее могли появляться старые секреты. Не повторять их; при
  сомнении перевыпускать.
- Перед изменением Nginx выполнять `nginx -t`, перед Docker-обновлением —
  `docker compose config --quiet`.
- Перед рискованной миграцией создавать свежий локальный и внешний снимок.
- Не удалять пользовательские файлы и не выполнять destructive Git-команды.
- Коммиты должны содержать только относящиеся к этапу изменения.
