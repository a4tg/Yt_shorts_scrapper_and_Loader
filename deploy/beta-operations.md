# Закрытая бета All As Planned

Этот регламент используется для первых 5–10 пользователей. Бета не запускается
массовой рекламой: каждый участник получает персональное приглашение, короткий
сценарий проверки и канал связи с владельцем продукта.

## Приглашение

1. Пользователь регистрируется сам и подтверждает email.
2. Владелец отправляет ссылку `https://allasplanned.ru/app`.
3. Пользователь проходит серверный блок «Быстрый старт» или создаёт
   идемпотентный демо-проект.
4. Для совместной проверки второй аккаунт сначала регистрируется, затем владелец
   пространства добавляет его как редактора, наблюдателя или клиента.
5. Пароли и токены не запрашиваются поддержкой и не передаются в чатах.

## Контрольный пользовательский сценарий

Участник без подсказок должен:

1. создать или открыть проект;
2. добавить карточку контента и дату публикации;
3. загрузить файл в медиатеку;
4. оставить комментарий и пройти согласование;
5. подготовить одно видео либо скачать исходник без оверлея;
6. найти результат в видеомедиатеке;
7. отправить тестовое обращение через раздел «Поддержка».

Успех фиксируется только по факту завершения сценария. Продуктовые события не
содержат текст карточек, сообщений, имена файлов или содержимое обращения.

## Работа с обращениями

- Новые обращения видны администратору SaaS и привязаны только к пользователю,
  workspace, проекту и разделу интерфейса.
- Ответ пользователю на закрытой бете отправляется вручную с
  `support@allasplanned.ru`.
- Текст обращения нельзя копировать в публичный issue или сторонний AI-сервис
  без удаления персональных и проектных данных.
- Статусы: `open` → `in_progress` → `resolved` или `closed`.
- Для решения указывается короткая заметка без секретов; она видна автору.

## Критический инцидент

Критическим считается утечка данных между workspace, невозможность входа,
повторное списание, потеря файла или остановка очереди.

1. Остановить затронутый сценарий, но не удалять данные и логи.
2. Проверить `/api/health/ready`, Telegram-монитор и журналы контейнера.
3. Зафиксировать request id, время, пользователя и затронутую сущность.
4. При риске утечки временно закрыть внешний доступ через nginx.
5. Создать резервную копию перед исправлением.
6. Проверить исправление тестом, затем полным release gate.
7. Сообщить затронутым участникам факт, период и принятые меры.

## Еженедельный разбор

- приглашены / подтвердили email;
- активны за последние 7 дней;
- завершили быстрый старт;
- дошли до готового материала;
- открытые и решённые обращения;
- ошибки очереди и платежей;
- причины, по которым пользователь не завершил сценарий.

До подтверждения серверных лимитов бета остаётся ограниченной 5–10 участниками.
